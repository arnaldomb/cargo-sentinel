2. Criar `apps/web/auth.ts` (config completa com Credentials + bcryptjs — só importado em Node runtime):
```typescript
import NextAuth from 'next-auth';
import Credentials from 'next-auth/providers/credentials';
import bcryptjs from 'bcryptjs';
import { z } from 'zod';
import { prisma } from '@cargo-sentinel/database';
import { authConfig } from './auth.config';

const loginSchema = z.object({
  email: z.string().email(),
  password: z.string().min(1),
});

export const { handlers, auth, signIn, signOut } = NextAuth({
  ...authConfig,
  providers: [
    Credentials({
      credentials: { email: {}, password: {} },
      async authorize(credentials) {
        const parsed = loginSchema.safeParse(credentials);
        if (!parsed.success) return null;
        const { email, password } = parsed.data;

        const user = await prisma.user.findUnique({
          where: { email },
          include: { empresa: { select: { status: true } } },
        });
        if (!user) return null;
        if (user.empresa && user.empresa.status === 'SUSPENSO') return null; // TENANT-01

        const valid = await bcryptjs.compare(password, user.passwordHash);
        if (!valid) return null;

        return {
          id: user.id,
          email: user.email,
          name: user.nome,
          role: user.role,
          empresaId: user.empresaId,
        };
      },
    }),
  ],
});
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser → /login | entrada não confiável (email/senha) cruza para o server action |
| Next.js Edge middleware → Node runtime | bcryptjs/Prisma só podem rodar no Node runtime, nunca no Edge |
| AUTH_SECRET | segredo compartilhado entre web e api — vazamento permite forjar tokens |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04 | Spoofing | POST /login (credential stuffing/brute force) | mitigate | rate limiting é aplicado no endpoint Express de login no Plan 03; aqui valida via bcryptjs.compare (resistente a timing por design do bcrypt) |
| T-02-05 | Elevation of Privilege | callback jwt role/empresaId | mitigate | role e empresaId vêm SOMENTE do banco via authorize(); cliente nunca fornece esses valores |
| T-02-06 | Spoofing | login de empresa suspensa | mitigate | authorize() rejeita se empresa.status === 'SUSPENSO' (TENANT-01) |
| T-02-07 | Information Disclosure | passwordHash | mitigate | select não retorna passwordHash ao cliente; authorize() descarta o hash após compare |
| T-02-08 | Tampering | JWE session token | mitigate | Auth.js criptografa o token (AES-256-CBC-HS512) com AUTH_SECRET; httpOnly cookie por padrão |
| T-02-09 | Denial of Service | redirect loop em /login | mitigate | matcher do middleware exclui /login e /api/auth (Pitfall 6) |
</threat_model>
