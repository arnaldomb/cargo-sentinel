<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP → /api/admin/* | Qualquer requisição autenticada — role SUPER_ADMIN obrigatório |
| Impersonation token → apps/web | Token emitido pelo backend e usado pelo frontend como session token |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | Elevation of Privilege | requireRole no adminRouter | mitigate | `requireRole('SUPER_ADMIN')` aplicado como middleware de nível de router — qualquer outro role recebe 403 imediatamente |
| T-07-02 | Information Disclosure | GET /api/admin/empresas | mitigate | Rota atrás de `requireRole('SUPER_ADMIN')` — dados multi-tenant só visíveis ao SUPER_ADMIN |
| T-07-03 | Elevation of Privilege | POST /api/admin/empresas/:id/impersonate | mitigate | Token impersonation não é refreshável (TTL 15 min hardcoded, sem refresh token emitido); payload inclui `impersonatedBy` para auditoria |
| T-07-04 | Tampering | POST /api/admin/empresas/:id/impersonate — chave HKDF | mitigate | Usa o mesmo `AUTH_SECRET` e derivação HKDF de auth.ts — token não pode ser forjado sem o segredo |
| T-07-05 | Information Disclosure | POST /api/admin/empresas — passwordHash | mitigate | `passwordHash` excluído da resposta 201 — apenas `{ id, email, nome, role }` retornado |
| T-07-06 | Denial of Service | POST /api/admin/empresas em loop | accept | Rota de baixo volume (apenas SUPER_ADMIN usa) — rate limiting por IP via Traefik é suficiente |
</threat_model>
