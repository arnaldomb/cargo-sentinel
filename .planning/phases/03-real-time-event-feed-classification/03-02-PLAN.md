# 03-02 — Bootstrap de Socket.IO com autenticação e rooms por tenant

## Objective

Adicionar a infraestrutura de realtime no backend sem abrir superfície de vazamento entre tenants: um servidor Socket.IO acoplado ao processo Express, autenticação baseada no mesmo cookie JWT/JWE do Auth.js e rooms fixas por `empresaId`.

## Scope

- subir `http.Server` compartilhado entre Express e Socket.IO
- criar middleware de autenticação do socket
- derivar `empresaId` exclusivamente do token
- entrar automaticamente na room `empresa:{empresaId}`
- criar helpers de emissão tenant-safe

## Out Of Scope

- dashboard frontend
- histórico de mensagens
- adapter Redis multi-instância

## Context

Hoje o backend expõe apenas HTTP. A Fase 3 precisa de push server→client para novos eventos e reclassificações. O risco principal aqui é vazar eventos de uma empresa para outra, então o desenho deve nascer com isolamento por room e sem aceitar `empresaId` do browser.

## Interfaces

### Socket context

```typescript
interface SocketUser {
  id: string;
  role: 'SUPER_ADMIN' | 'ADMIN_EMPRESA' | 'OPERADOR';
  empresaId: string | null;
}
```

### Event names

```typescript
'feed:evento-novo'
'feed:placa-classificada'
'feed:camera-status'
```

### Broadcast rule

```typescript
io.to(`empresa:${empresaId}`).emit(...)
```

Nunca:

```typescript
io.emit(...)
```

## Tasks

### Task 1: Adaptar bootstrap do servidor

- editar `apps/api/src/index.ts`
- trocar `app.listen(...)` por `http.createServer(app)`
- inicializar `Server` do `socket.io`
- exportar uma instância/helper consumível pelo restante do backend

### Task 2: Criar middleware de auth do socket

- criar `apps/api/src/realtime/auth.ts`
- ler cookie `authjs.session-token` ou `__Secure-authjs.session-token`
- reutilizar o padrão de decriptação da Phase 2 com `jose.jwtDecrypt` + `@panva/hkdf`
- rejeitar conexão se `empresaId` estiver ausente

### Task 3: Criar módulo de realtime seguro

- criar `apps/api/src/realtime/server.ts`
- no `connection`, adicionar o socket autenticado à room `empresa:{empresaId}`
- anexar dados tipados do usuário ao socket
- exportar funções como `emitEventoNovo(empresaId, dto)` e `emitPlacaClassificada(empresaId, dto)`

### Task 4: Cobrir com testes

- teste de auth do socket aceita token válido com `empresaId`
- teste rejeita conexão sem token
- teste rejeita conexão com token sem `empresaId`
- teste assegura que o helper emite apenas para `empresa:{empresaId}`

## Verification

- `pnpm --filter @cargo-sentinel/api test -- --run src/realtime`
- `pnpm --filter @cargo-sentinel/api build`

## Acceptance Criteria

- toda conexão autenticada entra automaticamente em `empresa:{empresaId}`
- nenhuma API de emissão usa `io.emit()`
- o browser não escolhe a room
- conexão sem token válido é rejeitada

## Done

O backend passa a ter uma camada de realtime segura por tenant, pronta para receber os eventos do feed e as mutações de classificação da fase seguinte.
