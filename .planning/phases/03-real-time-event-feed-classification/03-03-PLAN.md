# 03-03 — Feed inicial por HTTP, status operacional e wiring de emissão

## Objective

Entregar os dados operacionais que o dashboard precisa consumir: feed inicial por REST, status online/offline de câmeras e emissão em tempo real dos eventos relevantes gerados pela ingestão e pela reclassificação.

## Scope

- endpoint paginado para feed de eventos
- endpoint de status operacional de câmeras
- DTOs estáveis para REST e Socket.IO
- emissão de `evento-novo`, `placa-classificada` e `camera-status`

## Out Of Scope

- renderização do dashboard
- popover de classificação
- auto-scroll do frontend

## Context

O dashboard não pode depender apenas do socket. Ele precisa carregar um estado inicial consistente por HTTP e depois receber incrementos em tempo real. Essa separação também facilita testes e recuperação de sessão.

## Interfaces

### Feed endpoint

```http
GET /api/eventos/feed?limit=50&cursor=<eventoId>
```

### Camera status endpoint

```http
GET /api/cameras/status
```

### Feed item DTO

```typescript
type FeedItem = {
  id: string;
  timestamp: string;
  placaId: string | null;
  placaNumero: string;
  classificacao: string;
  direcao: 'ENTRADA' | 'SAIDA' | null;
  thumbnailUrl: string | null;
  obra: { id: string; nome: string };
  camera: { id: string; codigoLpr: string };
};
```

## Tasks

### Task 1: Criar DTO builders

- criar `apps/api/src/realtime/dto.ts`
- encapsular mapeamento de `Evento` para `FeedItem`
- encapsular cálculo de status de câmera

### Task 2: Criar feed REST

- criar `apps/api/src/routes/eventos.ts`
- listar eventos do tenant via `req.tenantClient`
- incluir joins de `Obra`, `Camera` e `Placa`
- gerar `thumbnailUrl` por presigned URL quando houver `fotoGarageKey`
- ordenar por `timestamp desc`
- suportar cursor simples por `id` ou `timestamp + id`

### Task 3: Criar status REST de câmeras

- estender `apps/api/src/routes/cameras.ts` ou criar rota dedicada
- retornar lista de câmeras do tenant com `ultimoEventoEm` e `status`
- considerar `online` se último evento ocorreu nos últimos 5 minutos

### Task 4: Ligar emissões do backend

- no worker LPR, após persistir evento, emitir `feed:evento-novo`
- após atualização de classificação, emitir `feed:placa-classificada`
- ao recalcular o último evento da câmera, emitir `feed:camera-status`

### Task 5: Cobrir com testes

- teste do feed retorna 200 com shape esperado
- teste do feed não cruza tenant
- teste do status marca câmera online/offline corretamente
- teste dos emit helpers é disparado com `empresaId` correto

## Verification

- `pnpm --filter @cargo-sentinel/api test -- --run src/routes/eventos src/routes/cameras src/realtime`
- `pnpm --filter @cargo-sentinel/api build`

## Acceptance Criteria

- o dashboard consegue hidratar o feed inicial sem reload
- o feed inclui thumbnail, placa, obra, câmera, classificação, horário e direção
- o tenant vê apenas seus próprios eventos
- cada câmera expõe `online/offline` com timestamp do último evento
- novas leituras e reclassificações geram emissão tenant-scoped

## Done

O backend passa a servir o estado inicial do painel e a publicar mudanças incrementais, deixando a UI da Fase 3 limitada apenas à apresentação e interação.
